# 케이스 스터디: AI 개발 하네스로 팀 생산성을 구조적으로 높이기

> AI 코딩 도구를 "잘 쓰게 만드는 시스템"을 설계하고 오픈소스로 배포한 사례입니다.
> 여러 기업 AX 사례를 연구하며 도출한 방법론과, 실제로 구축·배포한 하네스 시스템을 소개합니다.

---

## 배경

### 문제 인식

AI 코딩 도구(Claude Code, Copilot 등)가 보급되면서, 많은 팀에서 공통적인 문제가 발생합니다:

- 도구를 **개인적으로 쓰는 사람은 있지만**, 팀 차원의 활용은 없음
- 같은 도구를 써도 **사람마다 결과 품질이 크게 다름**
- AI가 생성한 코드에 대한 **품질 기준이 없음**
- "AI 쓰면 빠르다"는 말만 있고, **어떻게 쓰면 빠른지 정리된 게 없음**

여러 기업의 AX(AI Transformation) 사례를 연구하면서, 이 문제의 핵심은 도구가 아니라 **워크플로우**에 있다는 결론에 도달했습니다.

### 연구 기반

이 프로젝트는 아래 사례와 방법론을 참고하여 설계했습니다:

- **KT**: AX Engagement 4주 프로그램, FDE(Field Engineer) 모델
- **LG U+**: Secure AX, Dual Run 구조로 기존 시스템과 공존 후 전환
- **GS건설**: Agent 기반 현장 업무 자동화
- **공통 원칙**: "도구 도입이 아니라 일하는 방식의 전환", "되돌릴 수 없는 구조를 만들어야 성공"

## 해결 접근: AI 개발 하네스

### 하네스(Harness)란?

AI 코딩 도구를 **안전하고 일관되게 사용하기 위한 구조적 프레임워크**입니다.
단순히 "AI를 써라"가 아니라, "이 순서로, 이 규칙 안에서, 이 품질 기준으로 써라"를 시스템화한 것입니다.

### 설계 원칙

AX 사례 연구에서 도출한 3가지 원칙을 하네스에 반영했습니다:

**1. 계획 → 승인 → 구현 → 검증**
```
기존: 요청 → 코딩 → 리뷰 → 수정 → 완료

하네스: 요청 → 계획 수립 → 승인 → 코딩 → 자동 품질 검사 → 리뷰 → 완료
```
- AI에게 "바로 코딩해줘"가 아니라 "먼저 계획을 세워줘"로 시작
- 계획을 사람이 승인한 뒤에만 구현 진행
- 이 구조가 AI 산출물의 품질 편차를 줄이는 핵심

**2. 자동화된 품질 게이트**
- 코드 변경 시 자동으로 코딩 규칙, 보안 패턴을 검사
- 사람이 아닌 시스템이 일관된 기준으로 피드백
- "AI가 만든 코드도 우리 팀 기준을 충족하는가?"를 자동으로 확인

**3. 되돌릴 수 없는 구조**
- 워크플로우가 시스템에 내장되어, 개인의 의지와 관계없이 유지
- 가이드라인 문서가 아니라 **실행 가능한 설정과 템플릿**으로 제공

### 구현 및 배포

이 원칙을 실제 시스템으로 구현하여 오픈소스로 배포했습니다:

**[dev_sys_template](https://github.com/HSUNEH/dev_sys_template)** — Claude Code 기반 자동화 개발 시스템

주요 구성:
- **계획 관리 시스템**: 3문서(계획서, 맥락 노트, 체크리스트) 자동 생성 + 승인 워크플로우
- **자동 품질 검사**: Hook 시스템으로 코드 변경 시 자동 검사 (코딩 표준, 보안, 에러 처리)
- **개발 매뉴얼**: 프로젝트별 코딩 규칙을 시스템에 내장
- **서브에이전트**: QA, 테스트, 기획을 전담하는 AI 에이전트 자동 위임

## AX 관점에서의 시사점

### 왜 "하네스"가 AX 성공의 열쇠인가

기업 AX 사례를 연구하면서 반복적으로 확인한 패턴:

| AX 실패 패턴 | 하네스 접근 |
|-------------|-----------|
| 도구만 배포하고 끝 | 워크플로우까지 시스템화 |
| "알아서 써라" | 구조화된 프로세스 제공 |
| 품질은 각자 알아서 | 자동 품질 게이트 |
| 한 달 뒤 사용률 급감 | 되돌릴 수 없는 구조 |
| 보안 우려로 사용 중단 | 보안 거버넌스 내장 |

### 비개발 직군으로의 확장

이 하네스 접근 방식은 개발 업무에 국한되지 않습니다:

- **보고서 작성**: 프롬프트 템플릿 + 검토 프로세스 = 하네스
- **데이터 분석**: 분석 워크플로우 + 보안 등급 체크 = 하네스
- **고객 응대**: 응답 템플릿 + 품질 기준 + 감사 로그 = 하네스

핵심은 동일합니다: **"어떻게 쓸 것인가"를 구조화하고, 그 구조를 시스템에 내장하는 것.**

## 교훈

### 1. 도구가 아니라 워크플로우를 바꿔야 한다
AI 도구를 단순히 설치해주는 것만으로는 효과가 제한적입니다. **"언제, 어떻게, 어떤 순서로 사용하는가"를 시스템화**했을 때 비로소 효과가 나타납니다.

### 2. 보안 거버넌스가 선행되어야 한다
보안 문제를 나중으로 미루면, "혹시 문제되면 어쩌지"가 사용 자체를 막습니다. 도입 초기에 보안 가이드라인을 확립하는 것이 중요합니다.

### 3. "쓰면 좋은 것"이 아니라 "안 쓸 수 없는 것"을 만들어야 한다
자동 품질 검사, 표준화된 워크플로우처럼 시스템에 내장된 변화는 개인의 의지와 관계없이 유지됩니다.

### 4. 오픈소스로 공유하면 임팩트가 커진다
하네스를 오픈소스로 배포하면, 각 기업이 자기 상황에 맞게 커스터마이징할 수 있습니다. 하나의 팀이 아니라 생태계 전체의 AX를 가속합니다.
