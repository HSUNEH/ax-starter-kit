# AI 보안 사고 대응 플레이북

> AI 사용 중 보안 사고가 발생했을 때, 이 문서를 따라 대응하세요. 사고 유형별 절차, 보고 템플릿, 재발 방지까지 다룹니다.

---

## 대응 원칙

1. **빠르게 차단, 나중에 분석** — 사고 확인 즉시 해당 채널을 차단합니다. 원인 분석은 차단 후에 합니다.
2. **과잉 대응이 과소 대응보다 낫다** — 판단이 어려우면 한 단계 높은 심각도로 대응하세요.
3. **기록이 먼저** — 모든 대응 과정을 타임라인으로 기록합니다. 기록 없는 대응은 사후 분석이 불가능합니다.

---

## 사고 유형 분류

### 유형 1: 민감정보 입력 사고

**상황**: 직원이 고객 개인정보, 기밀 문서, API 키 등 민감정보를 AI에 입력한 경우.

| 심각도 | 기준 | 예시 |
|--------|------|------|
| **낮음** | 1~2등급 데이터, Enterprise/API 플랜 | 사내 공지 내용을 웹 채팅에 입력 |
| **중간** | 2등급 데이터, 무료/개인 플랜 | 내부 보고서를 개인 ChatGPT에 입력 |
| **높음** | 3~4등급 데이터, 어떤 플랜이든 | 고객 개인정보, 미발표 재무 데이터 입력 |
| **긴급** | 법적 보호 대상 대량 유출 | 고객 DB 전체를 AI에 입력, 의료 기록 입력 |

**대응 절차:**

```
[발견] → [1. 즉시 차단] → [2. 보고] → [3. 영향 평가] → [4. 후속 조치] → [5. 재발 방지]
```

1. **즉시 차단** (발견 후 30분 이내)
   - [ ] 해당 대화/세션을 삭제합니다
   - [ ] 해당 AI 서비스의 데이터 삭제 요청을 제출합니다 (지원되는 경우)
   - [ ] 유출된 비밀번호/API 키가 있으면 **즉시 변경**합니다

2. **보고** (발견 후 1시간 이내)
   - [ ] 팀 보안 담당자에게 보고합니다
   - [ ] 심각도 "높음" 이상이면 CISO/DPO에게 즉시 에스컬레이션합니다
   - [ ] 개인정보 유출이 확인되면 개인정보보호 담당자에게 보고합니다

3. **영향 평가** (24시간 이내)
   - [ ] 유출된 데이터의 종류와 규모를 파악합니다
   - [ ] AI 서비스 약관을 확인합니다 (데이터 학습 여부, 보관 기간)
   - [ ] 데이터가 제3자에게 노출되었을 가능성을 평가합니다

4. **후속 조치**
   - [ ] 법적 신고 의무 확인 (개인정보보호법 제34조: 1,000명 이상 유출 시 24시간 내 신고)
   - [ ] 영향받은 정보주체에게 통지 필요 여부 판단
   - [ ] 해당 직원에게 보안 재교육 실시

5. **재발 방지**
   - [ ] 사고 원인 분석 (왜 민감정보가 입력되었는가?)
   - [ ] 기술적 방지책 검토 (DLP, 입력 필터 등)
   - [ ] 프로세스 개선 (워크플로우에서 민감정보가 AI에 들어가는 경로 차단)

---

### 유형 2: 프롬프트 인젝션 / 탈옥 사고

**상황**: 외부 공격자 또는 내부 사용자가 프롬프트 조작을 통해 AI 시스템의 가드레일을 우회한 경우.

**주요 시나리오:**
- 시스템 프롬프트를 노출시키는 공격
- AI가 허용되지 않은 정보를 출력하도록 유도
- AI 기반 챗봇/에이전트를 통해 내부 시스템에 비정상 접근

**대응 절차:**

1. **즉시 차단**
   - [ ] 해당 AI 서비스/에이전트를 일시 중단합니다
   - [ ] 외부 공개 AI 서비스(챗봇 등)라면 즉시 오프라인 전환
   - [ ] 공격에 사용된 프롬프트를 증거로 보존합니다

2. **영향 평가**
   - [ ] 시스템 프롬프트가 노출되었는가?
   - [ ] 내부 데이터가 유출되었는가?
   - [ ] AI를 통해 다른 시스템에 비정상 접근이 발생했는가?
   - [ ] 공격이 자동화되었는가? (단발 vs 반복 공격)

3. **복구**
   - [ ] 시스템 프롬프트를 변경합니다
   - [ ] 입력 검증 로직을 강화합니다 (알려진 공격 패턴 필터링)
   - [ ] AI 에이전트의 권한을 재검토합니다
   - [ ] 보안 테스트 후 서비스를 재개합니다

---

### 유형 3: AI 에이전트 오작동 사고

**상황**: AI 에이전트(자동 실행 도구)가 의도하지 않은 동작을 수행한 경우.

**주요 시나리오:**
- 코드 에이전트가 프로덕션 환경에 의도하지 않은 변경을 배포
- AI가 승인 없이 외부 API를 호출하거나 데이터를 전송
- 자동화된 AI 작업이 무한 루프에 빠져 비용 폭증

**대응 절차:**

1. **즉시 차단**
   - [ ] 에이전트를 즉시 중지합니다
   - [ ] 에이전트가 접근한 시스템/서비스의 상태를 확인합니다
   - [ ] 의도하지 않은 변경이 있으면 롤백합니다

2. **영향 평가**
   - [ ] 어떤 시스템이 영향을 받았는가?
   - [ ] 외부로 데이터가 전송되었는가?
   - [ ] 금전적 피해가 발생했는가? (API 비용, 인프라 비용 등)

3. **복구 및 방지**
   - [ ] 에이전트 권한을 **최소 권한 원칙**으로 재설정합니다
   - [ ] 고위험 작업에 대한 사람 승인 단계를 추가합니다
   - [ ] 비용/실행 횟수 제한(hard limit)을 설정합니다
   - [ ] 에이전트 실행 로그를 모니터링 시스템에 연동합니다

---

### 유형 4: AI 출력물 오용 사고

**상황**: AI가 생성한 허위/부정확한 정보가 검증 없이 외부에 발송된 경우.

**주요 시나리오:**
- AI가 생성한 허위 통계/인용이 포함된 보고서가 고객에게 전달
- AI가 작성한 법률 검토 의견이 검증 없이 계약에 반영
- AI가 생성한 코드에 보안 취약점이 포함된 채로 배포

**대응 절차:**

1. **즉시 대응**
   - [ ] 해당 산출물의 배포/발송을 중단합니다
   - [ ] 이미 발송된 경우 수신자에게 정정 통보합니다
   - [ ] 코드인 경우 해당 배포를 롤백합니다

2. **영향 평가**
   - [ ] 오류의 범위와 심각도를 파악합니다
   - [ ] 영향받은 이해관계자를 식별합니다
   - [ ] 법적/계약적 영향이 있는지 확인합니다

3. **재발 방지**
   - [ ] AI 출력물 검증 프로세스를 강화합니다
   - [ ] 고위험 산출물(법률, 재무, 의료)에 대한 **이중 검토** 의무화
   - [ ] "AI 생성물" 라벨링 정책을 수립합니다

## 사고 보고 템플릿

사고 발생 시 아래 양식을 작성하여 보안팀에 제출하세요.

```
AI 보안 사고 보고서

[기본 정보]
보고일시: YYYY-MM-DD HH:MM
보고자: [이름 / 소속 / 연락처]
사고 발견일시: YYYY-MM-DD HH:MM
사고 유형: [민감정보 입력 / 프롬프트 인젝션 / 에이전트 오작동 / 출력물 오용 / 기타]
심각도: [낮음 / 중간 / 높음 / 긴급]

[사고 내용]
AI 도구: [도구명, 플랜, 사용 방식(웹/API/에이전트)]
사고 요약: (무엇이 발생했는가, 2~3문장)
유출/영향 데이터: (데이터 종류, 건수, 등급)
영향 범위: (영향받은 시스템, 사람, 고객)

[대응 현황]
즉시 차단 조치: [완료 / 진행중 / 미실시]
  - 수행 내용:
보고 완료: [CISO / DPO / 법무 / 미보고]
추가 조치 필요 여부: [예 / 아니오]
  - 필요 조치:

[타임라인]
HH:MM - [이벤트 1]
HH:MM - [이벤트 2]
HH:MM - [이벤트 3]
```

## 사후 분석 (Post-Mortem) 가이드

사고 대응이 완료된 후, **72시간 이내**에 사후 분석을 실시합니다.

### 분석 구조

```
1. 사고 요약
   - 무엇이 발생했는가? (1문단)
   - 영향 범위와 심각도

2. 타임라인
   - 사고 발생 → 발견 → 차단 → 복구 전체 과정

3. 근본 원인 (5 Whys)
   - Why 1: 왜 사고가 발생했는가?
   - Why 2: 왜 그 원인이 존재했는가?
   - Why 3: ...
   - (근본 원인에 도달할 때까지)

4. 잘된 점 / 개선할 점
   - 대응 과정에서 효과적이었던 부분
   - 대응 과정에서 부족했던 부분

5. 재발 방지 조치 (Action Items)
   - [ ] [조치 1] — 담당: [이름], 기한: [날짜]
   - [ ] [조치 2] — 담당: [이름], 기한: [날짜]
```

### 사후 분석 원칙

- **비난하지 않는다** — 사람이 아닌 시스템/프로세스의 문제를 찾는다
- **"다음에 조심하자"는 대책이 아니다** — 기술적/구조적 방지책을 도출한다
- **Action Item은 담당자와 기한이 있어야 한다** — 그렇지 않으면 실행되지 않는다

| 심각도 | 보고 대상 | 보고 기한 | 조건 |
|--------|-----------|-----------|------|
| **낮음** | 팀 보안 담당자 | 24시간 이내 | 1~2등급 데이터, Enterprise 플랜 |
| **중간** | 보안팀 | 4시간 이내 | 2등급 데이터, 개인 플랜 사용 |
| **높음** | CISO + 법무팀 | 1시간 이내 | 3~4등급 데이터 유출, 외부 영향 |
| **긴급** | 경영진 + 외부 신고 | 즉시 | 대량 개인정보 유출, 법적 신고 의무 |

## 예방적 보안 조치

사고가 발생하기 **전에** 이 조치들을 수립해두세요.

### 기술적 조치

- **입력 필터링**: DLP 솔루션으로 민감정보 입력을 자동 차단
  - 주민등록번호, 카드번호, API 키 패턴 탐지
  - 사내 AI 프록시 구축 시 프록시 단에서 필터링
- **에이전트 권한 관리**: 최소 권한 원칙 적용
  - 읽기 전용 / 쓰기 허용 / 실행 허용을 명확히 분리
  - 프로덕션 환경 접근은 별도 승인 필수
- **모니터링**: 이상 사용 패턴 자동 탐지
  - 대량 데이터 입력, 비정상 시간대 사용, 반복적 에러

### 조직적 조치

- **정기 교육**: 분기 1회 AI 보안 교육 (신규 입사자 필수)
- **모의 훈련**: 연 1회 AI 보안 사고 대응 훈련
- **보안 챔피언**: 팀별 AI 보안 담당자 지정 (보안팀과 소통 창구)
