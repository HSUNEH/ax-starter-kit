# AX 4주 실행 플레이북

> 이 문서 하나를 따라가면, 4주 만에 팀에 AI를 정착시킬 수 있습니다.

---

## 시작 전 체크리스트

플레이북을 시작하기 전에 이것들이 준비되어 있어야 합니다:

- [ ] **경영진/리더 동의**: "AI 전환을 해보겠다"는 의사결정 (Week 0 자료 활용)
- [ ] **AX 담당자 지정**: 이 플레이북을 실행할 사람 (1~2명)
- [ ] **대상 팀 선정**: 처음 적용할 팀 (5~15명 규모 권장)
- [ ] **AI 도구 선정**: 팀이 사용할 AI 도구 결정 (ChatGPT, Claude, Copilot 등)
- [ ] **보안 사전 승인**: IT/보안팀과 AI 사용 범위 합의 ([보안 가이드](03-security/) 참조)

## Week 0: 왜 AX를 해야 하는가

> 도구를 도입하기 전에, "왜"에 대한 공감대가 먼저입니다.

### AX는 "AI 도입"이 아니다

많은 기업이 AI 전환을 "ChatGPT 계정을 만들어주는 것"으로 시작합니다.
그리고 한 달 뒤, 아무도 쓰지 않습니다.

**AX(AI Transformation)는 AI 도구 도입이 아니라, 일하는 방식의 전환입니다.**

| | AI 도입 (실패 패턴) | AX (성공 패턴) |
|---|---|---|
| 접근 | "이 도구를 써보세요" | "이 업무를 이렇게 바꿉시다" |
| 주체 | IT팀이 배포 | **리더가 직접 사용하고 전파** |
| 범위 | 개인 선택 | **팀 단위 워크플로우 변경** |
| 지속성 | 신기함이 사라지면 중단 | **되돌릴 수 없는 구조로 정착** |
| 측정 | "써보니까 좋더라" | **절감 시간/비용을 수치로 증명** |

### 안 하면 어떻게 되는가

이건 "하면 좋은 것"이 아니라, **안 하면 뒤처지는 것**입니다.

**생산성 격차는 이미 벌어지고 있습니다:**
- AI를 업무에 활용하는 직원은 그렇지 않은 직원 대비 **2~3배의 업무 처리량** 차이를 보입니다
- 국내 대기업(LG전자, KT, 삼성전자, GS건설)은 이미 조직 단위 AX를 추진 중입니다
- 이 격차는 개인이 아닌 **조직 단위**로 벌어집니다 — 한 명이 잘해도 팀이 안 쓰면 효과가 제한적입니다

**경쟁사는 이미 움직이고 있습니다:**
- KT: AX Engagement 4주 프로그램, FDE(Field Engineer) 모델 도입
- LG U+: Secure AX 5가지 여정, Dual Run 구조로 기존 시스템과 공존 후 전환
- GS건설: 작업 지시 Agent, 출역일보 Agent로 현장 업무 자동화

### 흔한 오해

> **"우리 업종은 AI랑 거리가 멀다"**
> → AI는 코딩만 하는 게 아닙니다. 보고서 작성, 이메일, 회의록 정리, 데이터 분석 — 모든 사무 업무에 적용됩니다. 업종이 아니라 **업무 유형**이 기준입니다.

> **"도구만 도입하면 알아서 쓰겠지"**
> → 안 씁니다. 엑셀이 있어도 수기로 계산하는 사람이 있는 것과 같습니다. **워크플로우 자체를 바꿔야** 합니다.

> **"AI가 내 일을 대체하는 거 아닌가"**
> → AI는 사람을 대체하는 게 아니라 **역량을 확장**하는 도구입니다. AI를 쓰는 사람이 안 쓰는 사람을 대체합니다.

> **"보안이 걱정된다"**
> → 맞습니다. 그래서 [보안 가이드](03-security/)가 있습니다. API 호출 방식, 데이터 등급별 사용 범위, Enterprise 계약 등 해결 방법이 있습니다.

### 경영진 설득용 한 페이지 요약

아래 내용을 복사해서 경영진 보고에 사용하세요:

---

**AI 전환(AX) 제안 요약**

**현황**: AI 도구가 존재하지만, 조직적 활용은 부재. 개인 사용에 그치고 있음.

**제안**: 4주간 시범 팀(00팀, 00명)에 AI 워크플로우를 도입하고 성과를 측정.

**기대 효과**:
- 반복 업무(보고서, 이메일, 데이터 정리) 처리 시간 30~50% 절감
- 팀 단위 AI 활용 문화 정착

**필요 자원**:
- AX 담당자 1명 (기존 업무와 병행 가능)
- AI 도구 라이선스 비용 (월 $20~60/인)
- 4주간 주 2시간의 팀 미팅 시간

**리스크 관리**:
- 보안 가이드라인 사전 수립 (데이터 등급별 사용 범위 정의)
- 기존 시스템과 병행 운영 (Dual Run), 전환 아닌 통합 방식

**성공 기준**: 4주 후 대상 팀의 주간 반복 업무 시간 20% 이상 절감

## Week 1: 현황 진단 + AI 기초 체험

> 우리 팀이 뭘 하고 있는지 파악하고, 핵심 인원이 먼저 체험합니다.

### Day 1-2: 업무 프로세스 매핑

팀원 전체에게 아래 질문으로 업무를 수집합니다:

1. **매일 반복하는 업무**는 무엇인가? (예: 이메일 답변, 보고서 작성, 데이터 입력)
2. 그 업무에 **얼마나 시간**을 쓰는가?
3. 그 업무에서 **가장 귀찮은 부분**은?

> 💡 **자동화 우선순위 원칙**: (1) 많은 사람이 공통으로 매일 하는 업무 (2) 노하우가 섞여 자동화 효과가 큰 업무

수집한 업무를 [AI 적용 가능성 체크리스트](01-diagnosis/ai-readiness-checklist.md)로 평가합니다.

### Day 3: AI 적용 우선순위 선정

수집된 업무 중 **2~3개를 선정**합니다. 선정 기준:

| 기준 | 점수 (1-5) |
|------|-----------|
| 빈도 (매일 하는가?) | |
| 인원 (몇 명이 하는가?) | |
| 시간 (한 번에 얼마나 걸리는가?) | |
| AI 적합성 (텍스트/데이터 기반인가?) | |

총점이 높은 업무 2~3개를 Week 2에서 워크플로우로 설계합니다.

### Day 4-5: 핵심 인원 AI 체험

리더 + 핵심 인원 3~5명이 **직접** AI를 사용해봅니다.

**왜 리더부터인가?**
> 리더가 AI를 직접 사용해야 조직에 전파됩니다. 리더가 전환되면, 머릿속 노하우가 AI를 통해 조직 자산으로 축적됩니다.

**체험 방법:**
1. 각자 본인의 실제 업무 1개를 골라 AI로 수행
2. [비개발자 온보딩 가이드](02-onboarding/non-dev/getting-started.md)를 따라 진행
3. 결과를 팀에 공유 (잘된 점, 아쉬운 점)

**이번 주 산출물:**
- [ ] 업무별 AI 적용 가능성 평가표
- [ ] AI 적용 우선순위 업무 2~3개 선정
- [ ] 핵심 인원 체험 완료 및 피드백

## Week 2: 업무별 AI 워크플로우 설계

> 선정된 업무에 대해 "AI를 어떻게 쓸 것인가"를 구체적으로 설계합니다.

### 워크플로우 설계 방법

선정된 업무마다 아래를 작성합니다:

**1. Before (현재 방식)**
```
[트리거] → [수동 작업 1] → [수동 작업 2] → ... → [산출물]
예: 회의 끝남 → 메모 정리 → 문장 다듬기 → 공유 (30분)
```

**2. After (AI 적용 방식)**
```
[트리거] → [AI 작업] → [사람 검토] → [산출물]
예: 회의 끝남 → AI에 메모 붙여넣기 → 검토/수정 → 공유 (5분)
```

**3. 프롬프트 템플릿**
해당 업무에 맞는 프롬프트를 작성합니다.
[프롬프트 템플릿 모음](02-onboarding/non-dev/)에서 기본형을 가져와 팀 맞춤으로 수정하세요.

### 커스터마이징 팁

- 프롬프트에 **팀/회사 맥락**을 넣을수록 품질이 올라갑니다
  - "우리 팀은 B2B SaaS 회사입니다" → AI가 맥락에 맞는 톤으로 작성
- **산출물 형식을 지정**하세요
  - "표 형태로", "불릿포인트 3개로", "슬랙에 올릴 수 있는 길이로"
- Before/After를 **반드시 기록**하세요 — Week 4 성과 측정에 쓰입니다

### 보안 체크

워크플로우에 들어가는 데이터의 보안 등급을 확인합니다.
[데이터 등급별 AI 사용 가이드](03-security/data-classification.md)를 참조하세요.

**이번 주 산출물:**
- [ ] 업무별 Before/After 워크플로우 문서
- [ ] 팀 맞춤 프롬프트 템플릿 2~3개
- [ ] 보안 등급 확인 완료

## Week 3: 팀 전체 적용 + 습관화

> 핵심 인원에서 팀 전체로 확산합니다. 핵심은 "습관"입니다.

### 팀 온보딩 워크숍 (1~2시간)

**구성:**
1. **왜 하는가** (10분) — Week 0 내용 요약
2. **데모** (20분) — 핵심 인원이 실제 업무에서 AI를 쓰는 과정 시연
3. **함께 해보기** (30분) — 각자 본인 업무에 프롬프트 템플릿 적용
4. **Q&A** (15분) — 불안감, 보안, 품질 관련 질문 해소

> 💡 **포인트**: 이론 설명을 최소화하고, "직접 해보기"에 시간을 많이 할당하세요. 5분 써보는 것이 30분 설명보다 낫습니다.

### 챔피언 지정

각 소그룹(3~5명)에서 **챔피언 1명**을 지정합니다.

챔피언의 역할:
- 주변 동료가 AI를 처음 쓸 때 옆에서 도와주기
- 좋은 프롬프트나 사용법을 팀에 공유
- 안 쓰는 동료를 1:1로 케어

> 챔피언은 기술 전문가가 아니어도 됩니다. **"AI를 열심히 쓰고, 남에게 알려주는 걸 좋아하는 사람"**이면 충분합니다.

### 습관화 장치

**일일 체크인 (슬랙/팀즈):**
- 매일 퇴근 전 한 줄: "오늘 AI로 한 일: ___"
- 부담 없는 수준으로 유지 (강제 아닌 권장)

**주간 공유:**
- 주 1회 15분 팀 미팅에서 "이번 주 AI 사용 Best/Worst"를 공유
- 잘 된 사례를 팀 프롬프트 라이브러리에 추가

**안 쓰는 사람 케어:**
- 2주차까지 AI를 한 번도 안 쓴 사람이 있으면 챔피언이 1:1로 첫 사용을 도와줌
- **절대 강제하지 않는다** — "한번 같이 해볼까요?"로 접근

### "되돌릴 수 없는 변화" 만들기

> AX의 핵심: 한 달 뒤에도 AI를 계속 쓰려면, **되돌릴 수 없는 구조**를 만들어야 합니다.

**방법:**
- AI 산출물을 **공식 프로세스에 통합** (예: 회의록은 AI로만 작성, 기존 양식 폐기)
- 프롬프트 템플릿을 **팀 공유 문서에 고정** (노션, 컨플루언스 등)
- 절감된 시간 일부를 **학습에 사용 보장** (공식적으로)

**이번 주 산출물:**
- [ ] 팀 온보딩 워크숍 완료
- [ ] 챔피언 지정 완료
- [ ] 일일 체크인 채널 개설
- [ ] 1개 이상의 업무에서 AI 산출물이 공식 프로세스에 통합

## Week 4: 성과 측정 + 지속 구조 수립

> 수치로 증명하고, 멈추지 않는 구조를 만듭니다.

### 성과 측정

**정량 지표:**

| 지표 | 측정 방법 | Before | After |
|------|-----------|--------|-------|
| 업무 처리 시간 | 대상 업무 2~3개의 평균 소요 시간 | | |
| AI 활용률 | 팀원 중 주 1회 이상 AI 사용 비율 | 0% | |
| 산출물 품질 | 수정 횟수, 피드백 건수 비교 | | |

**정성 지표:**
- 팀원 만족도 설문 (5점 척도)
- "AI가 가장 도움이 된 업무"
- "개선이 필요한 점"

### 성과 리포트 작성

아래 구조로 경영진/유관부서에 보고합니다:

```
1. 목적: 왜 이 시범을 했는가
2. 대상: 누가, 어떤 업무에 적용했는가
3. 결과: 정량/정성 지표
4. 교훈: 뭐가 잘 됐고, 뭐가 안 됐는가
5. 제안: 확산 계획 (다음 팀, 다음 업무)
```

### 지속 구조 수립

**정기 리뷰:**
- 월 1회 "AI 활용 리뷰" (30분)
- 새로운 업무 적용 논의, 프롬프트 업데이트

**프롬프트 라이브러리:**
- 팀에서 검증된 프롬프트를 모아 공유 문서화
- 신규 입사자 온보딩 자료에 포함

**확산 계획:**
- 성과가 확인되면 다음 팀으로 확대
- 이 플레이북을 다음 팀 AX 담당자에게 전달

**이번 주 산출물:**
- [ ] Before/After 성과 비교 완료
- [ ] 팀원 피드백 수집 완료
- [ ] 성과 리포트 작성
- [ ] 지속 운영 계획 수립
- [ ] (선택) 다음 팀 확산 계획

## 실패하는 AX의 패턴

마지막으로, 실전에서 겪은 실패 패턴입니다. 이걸 피하세요:

| 패턴 | 증상 | 해결 |
|------|------|------|
| **도구만 던져줌** | 계정만 만들어주고 끝. 한 달 뒤 사용률 5% | 워크플로우까지 바꿔야 함 (이 플레이북 전체) |
| **리더가 안 씀** | "너희들이나 써봐". 팀원도 안 씀 | 리더가 먼저 체험하고 시연해야 함 (Week 1) |
| **보안 미합의** | "혹시 문제되면 어쩌지" → 아예 안 씀 | 사전에 보안 가이드라인 합의 (시작 전 체크리스트) |
| **성과 측정 안 함** | "좋은 것 같은데..." → 예산 삭감 | 정량 지표를 처음부터 설계 (Week 4) |
| **강제 도입** | "다 써야 합니다" → 반발 | 챔피언 기반 자발적 확산 (Week 3) |
| **일회성 이벤트** | 워크숍 한 번 하고 끝 | 되돌릴 수 없는 구조 만들기 (Week 3) |
