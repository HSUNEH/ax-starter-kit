# 개발 워크플로우 템플릿

> 새 기능 개발, 버그 수정, 리팩토링 — 각 상황에서 AI를 어떤 순서로 활용하면 좋은지 정리한 워크플로우입니다.

---

## 사용 방법

1. 현재 작업에 맞는 워크플로우를 선택합니다
2. 각 단계를 순서대로 따릅니다
3. AI 프롬프트 예시를 복사하여 사용합니다
4. 체크리스트로 빠뜨린 단계가 없는지 확인합니다

---

## 워크플로우 1: 새 기능 개발

새로운 기능을 처음부터 구현할 때 사용합니다.

### 전체 흐름

```
[1. 설계] → [2. 구현] → [3. 테스트] → [4. 리뷰] → [5. 문서화]
```

### 1단계: 설계

AI에게 요구사항을 주고, 구현 방향을 먼저 정합니다.

```
아래 요구사항을 구현하려고 합니다. 코드를 작성하기 전에 설계를 먼저 제안해줘.

[요구사항]
- (기능 설명)

[기술 스택]
- (프레임워크, 언어 등)

제안해줄 내용:
1. 필요한 파일/모듈 목록
2. 핵심 데이터 구조
3. 주요 로직 흐름
4. 고려해야 할 엣지 케이스
```

> **핵심**: 코드부터 작성하지 마세요. 설계를 먼저 리뷰하면 나중에 대규모 수정을 피할 수 있습니다.

### 2단계: 구현

설계가 확정되면, 단위별로 나눠서 구현합니다.

```
위 설계를 바탕으로 [모듈/파일명]을 구현해줘.

제약:
- (코딩 컨벤션)
- (기존 패턴 참조 파일)
- (사용 금지 패턴)
```

**구현 순서 팁:**
1. 데이터 모델/타입 정의 먼저
2. 핵심 비즈니스 로직
3. API/인터페이스 레이어
4. UI 컴포넌트 (프론트엔드인 경우)

### 3단계: 테스트

구현이 끝나면 테스트를 작성합니다.

```
방금 구현한 [기능]에 대한 테스트를 작성해줘.

테스트 범위:
- 정상 케이스: (주요 시나리오)
- 에러 케이스: (실패해야 하는 시나리오)
- 엣지 케이스: (경계값, 빈 입력 등)

테스트 프레임워크: [Jest/Vitest/pytest 등]
기존 테스트 패턴: [기존 테스트 파일 경로]
```

### 4단계: 리뷰

커밋 전에 AI로 셀프 리뷰합니다.

```
아래 변경사항을 리뷰해줘.

중점 확인:
- 보안 취약점
- 에러 핸들링 누락
- 성능 이슈
- 기존 코드와의 일관성

(git diff 결과 또는 코드)
```

### 5단계: 문서화

```
방금 구현한 [기능]에 대한 문서를 작성해줘.
- 기능 설명 (1~2문장)
- 사용법 (코드 예시)
- API 스펙 (해당 시)
```

### 체크리스트

- [ ] 설계 리뷰 완료
- [ ] 핵심 로직 구현 완료
- [ ] 테스트 작성 및 통과
- [ ] 셀프 코드 리뷰 완료
- [ ] 필요한 문서 작성 완료
- [ ] 보안 체크 (민감정보, 인증, 입력 검증)
- [ ] PR 생성

## 워크플로우 2: 버그 수정

기존 코드의 버그를 수정할 때 사용합니다.

### 전체 흐름

```
[1. 분석] → [2. 재현] → [3. 수정] → [4. 검증] → [5. 회귀 방지]
```

### 1단계: 분석

AI에게 에러 정보를 주고 원인을 파악합니다.

```
아래 버그를 분석해줘.

[증상]
- (무엇이 잘못되고 있는가)

[에러 로그]
(에러 메시지 / 스택 트레이스)

[재현 조건]
- (어떤 상황에서 발생하는가)

원인을 분석하고, 수정 방안을 제안해줘.
코드를 수정하기 전에 분석 결과부터 보여줘.
```

> **핵심**: AI에게 바로 "고쳐줘"라고 하지 마세요. 원인 분석을 먼저 확인해야 엉뚱한 곳을 수정하는 걸 방지합니다.

### 2단계: 재현

분석 결과를 바탕으로 재현 테스트를 작성합니다.

```
위 분석을 바탕으로, 이 버그를 재현하는 테스트를 작성해줘.
- 현재 상태에서는 실패해야 함 (버그 재현)
- 수정 후에는 통과해야 함
```

### 3단계: 수정

원인이 확인되면 최소한의 수정만 합니다.

```
위 분석대로 버그를 수정해줘.

제약:
- 버그 수정에 필요한 최소한의 변경만
- 관련 없는 리팩토링은 하지 않기
- 기존 동작을 변경하지 않기
```

### 4단계: 검증

```
수정한 코드가 올바른지 확인해줘.
1. 재현 테스트가 통과하는가?
2. 기존 테스트가 모두 통과하는가?
3. 수정이 다른 곳에 영향을 미치지 않는가?
```

### 5단계: 회귀 방지

```
이 버그가 다시 발생하지 않도록:
1. 재현 테스트를 테스트 스위트에 추가해줘
2. 같은 패턴의 코드가 다른 곳에도 있는지 확인해줘
```

### 체크리스트

- [ ] 근본 원인 파악 완료
- [ ] 재현 테스트 작성
- [ ] 최소 범위 수정 완료
- [ ] 재현 테스트 통과
- [ ] 기존 테스트 전부 통과
- [ ] 동일 패턴 다른 위치 확인
- [ ] PR 생성 (원인 + 수정 내용 기술)

## 워크플로우 3: 리팩토링

기존 코드의 구조를 개선할 때 사용합니다.

### 전체 흐름

```
[1. 현황 분석] → [2. 계획] → [3. 안전장치] → [4. 실행] → [5. 검증]
```

### 1단계: 현황 분석

```
아래 코드/모듈의 현재 상태를 분석해줘.

대상: [파일 경로 또는 모듈명]

분석해줄 항목:
1. 현재 구조의 문제점
2. 복잡도가 높은 부분
3. 중복 코드
4. 의존 관계
```

### 2단계: 계획

```
위 분석을 바탕으로 리팩토링 계획을 세워줘.

목표: [구체적 개선 목표]

제약:
- 외부 인터페이스(API, 함수 시그니처)는 유지
- 새 라이브러리 추가 금지
- 한 번에 하나의 변경만 (작은 단위로 나눠서)

제안 형식:
1. [변경 1] - 무엇을, 왜
2. [변경 2] - 무엇을, 왜
3. ...
```

> **핵심**: 리팩토링은 반드시 **작은 단위로 나눠서** 진행하세요. 하나의 거대한 리팩토링 PR은 리뷰가 불가능합니다.

### 3단계: 안전장치

리팩토링 전에 테스트 커버리지를 확보합니다.

```
리팩토링 대상 코드에 테스트가 충분한지 확인해줘.

대상: [파일 경로]

- 기존 테스트가 있는가?
- 커버리지가 충분한가?
- 리팩토링 전에 추가해야 할 테스트가 있는가?

(부족하면) 리팩토링 전에 현재 동작을 보존하는 테스트를 먼저 작성해줘.
```

### 4단계: 실행

계획된 순서대로 하나씩 실행합니다.

```
리팩토링 계획의 [N단계]를 실행해줘.

이 단계의 변경:
- [구체적 변경 내용]

확인:
- 기존 테스트 통과 여부
- 외부 인터페이스 유지 여부
```

### 5단계: 검증

```
리팩토링이 완료되었습니다. 최종 검증해줘.

1. 모든 테스트가 통과하는가?
2. 외부 인터페이스가 변경되지 않았는가?
3. 코드 복잡도가 실제로 줄었는가? (Before/After 비교)
4. 의도하지 않은 동작 변경이 없는가?
```

### 체크리스트

- [ ] 현황 분석 완료
- [ ] 리팩토링 계획 수립 (작은 단위로 분할)
- [ ] 기존 테스트 커버리지 확보
- [ ] 단계별 리팩토링 실행
- [ ] 각 단계 후 테스트 통과 확인
- [ ] 최종 Before/After 비교
- [ ] PR 생성 (변경 이유와 영향 범위 기술)

## 워크플로우 4: 코드 리뷰

다른 팀원의 PR을 AI와 함께 리뷰할 때 사용합니다.

### 전체 흐름

```
[1. 변경 파악] → [2. AI 프리뷰] → [3. 직접 리뷰] → [4. 피드백 작성]
```

### 1단계: 변경 파악

```
이 PR의 변경 내용을 요약해줘.

(git diff 또는 변경된 파일 목록)

요약:
- 무엇이 변경되었는가 (1~2문장)
- 영향받는 기능/모듈
- 변경 크기 (작음/중간/큼)
```

### 2단계: AI 프리뷰

```
이 변경사항을 리뷰해줘.

중점 항목:
1. [보안] 인증 우회, 인젝션, XSS 가능성
2. [정합성] 기존 코드와의 일관성
3. [에러 처리] 누락된 에러 핸들링
4. [성능] N+1 쿼리, 불필요한 연산
5. [테스트] 테스트 커버리지 충분 여부

각 이슈에 [심각도]와 [위치]를 표시해줘.
```

### 3단계: 직접 리뷰

AI의 리뷰를 참고하되, **반드시 직접 확인**합니다:

- AI가 놓칠 수 있는 것: 비즈니스 로직의 정확성, 요구사항 충족 여부
- AI가 잘 찾는 것: 패턴 불일치, 보안 취약점, 에러 핸들링 누락

### 4단계: 피드백 작성

```
아래 리뷰 포인트들을 PR 코멘트용으로 정리해줘.

리뷰 포인트:
1. (이슈 1)
2. (이슈 2)

톤: 건설적이고 구체적으로 (무엇이 문제이고, 어떻게 수정하면 좋은지)
```

## 워크플로우 선택 가이드

| 상황 | 워크플로우 | 핵심 주의점 |
|------|-----------|------------|
| 새 API 엔드포인트 추가 | 새 기능 개발 | 설계 먼저, 코드 나중 |
| 고객이 에러를 보고함 | 버그 수정 | 원인 분석 먼저, 수정 나중 |
| "이 코드 너무 복잡해" | 리팩토링 | 테스트 먼저, 리팩토링 나중 |
| PR이 올라옴 | 코드 리뷰 | AI 참고, 직접 판단 |
| 새 프로젝트 시작 | 새 기능 개발 + 하네스 | [하네스 가이드](harness-guide.md) 참조 |

---

*이 워크플로우 템플릿과 [프롬프트 엔지니어링 가이드](prompt-engineering.md)를 함께 사용하면 더 효과적입니다. [하네스](harness-guide.md)를 사용하면 매번 맥락을 입력하지 않아도 됩니다.*
