# AI 개발 하네스 도입 가이드

> AI 코딩 도구를 "그냥 쓰는 것"과 "체계적으로 쓰는 것"은 다릅니다. 하네스는 팀 전체가 일관된 품질로 AI를 활용하도록 만드는 구조입니다.

---

## 하네스란?

AI 개발 하네스(harness)는 AI 코딩 에이전트에게 **프로젝트의 규칙, 워크플로우, 품질 기준을 미리 알려주는 설정 체계**입니다.

### 하네스 없이 AI를 쓰면

```
개발자 A: "컴포넌트 만들어줘" → 클래스형 컴포넌트, CSS Modules
개발자 B: "컴포넌트 만들어줘" → 함수형 컴포넌트, Tailwind
개발자 C: "컴포넌트 만들어줘" → 함수형 컴포넌트, styled-components
→ 같은 프로젝트에 3가지 스타일이 공존
```

### 하네스를 쓰면

```
하네스 규칙: "함수형 컴포넌트 + Tailwind + 네이밍은 PascalCase"
개발자 A: "컴포넌트 만들어줘" → 함수형, Tailwind, PascalCase ✅
개발자 B: "컴포넌트 만들어줘" → 함수형, Tailwind, PascalCase ✅
개발자 C: "컴포넌트 만들어줘" → 함수형, Tailwind, PascalCase ✅
→ 팀 전체가 동일한 품질과 스타일
```

### 하네스의 핵심 구성 요소

| 구성 요소 | 역할 | 비유 |
|-----------|------|------|
| **프로젝트 설정 (CLAUDE.md 등)** | AI에게 프로젝트 규칙을 알려줌 | 신입 개발자에게 주는 온보딩 문서 |
| **워크플로우 규칙** | 계획→승인→구현→검증 순서를 강제 | 코드 리뷰 필수 정책 |
| **Hook (자동 검사)** | 코드 변경 시 자동으로 품질 검사 | CI/CD 파이프라인 |
| **서브에이전트** | 특정 작업을 전문화된 AI에게 위임 | QA팀, 설계팀 역할 분담 |
| **프롬프트 템플릿** | 반복 작업을 표준화된 명령으로 | 스크립트/매크로 |

## 왜 하네스가 필요한가

### 1. 일관성

개인마다 다른 프롬프트를 쓰면 결과물 품질이 들쭉날쭉합니다. 하네스는 **팀 전체의 AI 결과물을 표준화**합니다.

### 2. 안전장치

AI가 프로덕션 코드를 직접 수정하거나, 승인 없이 배포하는 상황을 방지합니다. 하네스의 워크플로우 규칙이 **계획→승인→구현 순서를 강제**합니다.

### 3. 지식 축적

개인의 프롬프트 노하우가 하네스 설정에 녹아들어 **팀 자산으로 축적**됩니다. 사람이 바뀌어도 AI 활용 수준이 유지됩니다.

### 4. 점진적 확장

처음에는 간단한 설정으로 시작하고, 팀의 숙련도가 올라가면 Hook, 서브에이전트 등을 추가합니다.

## dev_sys_template 소개

[**dev_sys_template**](https://github.com/your-org/dev_sys_template)은 Claude Code 기반 AI 개발 하네스 템플릿입니다. 복사해서 바로 사용할 수 있습니다.

### 구조

```
dev_sys_template/
├── cwm/
│   ├── CLAUDE.md            ← 프로젝트 규칙 (AI가 읽는 온보딩 문서)
│   ├── hooks/               ← 자동 품질 검사
│   │   └── config.yml       ← Hook 동작 규칙
│   ├── agents/              ← 서브에이전트 (QA, 테스트, 기획)
│   ├── skills/              ← 재사용 가능한 작업 템플릿
│   └── templates/           ← 계획서, 체크리스트 템플릿
```

### 핵심 워크플로우: 계획 → 승인 → 구현 → 검증

```
1. [계획] 새 작업 시작 → /plan-manager로 계획서 생성
   └── PLAN.md, CHECKLIST.md, CONTEXT.md 자동 생성

2. [승인] 계획서를 사람이 검토 → 승인/수정 요청
   └── 승인 전까지 코드 작성 금지

3. [구현] 승인 후 Phase별 순서대로 구현
   └── CHECKLIST.md를 실시간 업데이트

4. [검증] Hook이 코드 변경을 자동 검사
   └── 0건=통과, 1~3건=즉시수정, 4건+=서브에이전트 위임
```

이 워크플로우가 중요한 이유: **AI가 검토 없이 코드를 마구 생성하는 것을 방지**합니다.

## 설치 및 세팅

### 전제 조건

- Claude Code 설치 완료 ([시작 가이드](getting-started.md) 참조)
- Git 사용 가능한 프로젝트

### 방법 1: 템플릿에서 새 프로젝트 시작

```bash
# 1. 템플릿 클론
git clone https://github.com/your-org/dev_sys_template.git my-project
cd my-project

# 2. Claude Code 시작
claude

# 3. 초기 설정 실행
/setup
# → 프로젝트 이름, 기술 스택, 규칙 등을 대화형으로 설정
```

### 방법 2: 기존 프로젝트에 하네스 추가

```bash
# 1. 기존 프로젝트로 이동
cd your-existing-project

# 2. 하네스 설정 파일 복사
# (dev_sys_template에서 .claude/ 디렉토리와 CLAUDE.md를 복사)

# 3. CLAUDE.md를 프로젝트에 맞게 수정
# → 기술 스택, 코딩 규칙, 빌드 명령어 등

# 4. Claude Code 시작
claude

# 5. /setup으로 프로젝트 맞춤 설정
/setup
```

### CLAUDE.md 커스터마이징

CLAUDE.md는 AI가 프로젝트에서 지켜야 할 규칙을 정의합니다. 최소한 아래 항목을 포함하세요:

```markdown
# CLAUDE.md

## 이 프로젝트
- **이름**: [프로젝트명]
- **설명**: [한 줄 설명]
- **기술 스택**: [React, TypeScript, Tailwind 등]

## 코딩 규칙
- [네이밍 컨벤션]
- [파일 구조 규칙]
- [금지 패턴]

## 주요 명령어
- 빌드: [npm run build 등]
- 테스트: [npm test 등]
- 린트: [npm run lint 등]
```

## 단계별 도입 전략

한 번에 모든 기능을 켜지 마세요. 팀의 AI 숙련도에 맞게 단계적으로 도입합니다.

### 1단계: CLAUDE.md만 (1일)

- 프로젝트 규칙과 코딩 컨벤션을 CLAUDE.md에 정리
- AI가 프로젝트 맥락을 이해하고 일관된 코드를 생성하는지 확인
- **이것만으로도 AI 결과물 품질이 크게 향상됩니다**

### 2단계: 워크플로우 규칙 추가 (1주)

- 계획→승인→구현 워크플로우 적용
- `/plan-manager`로 계획서를 생성하고, 승인 후 구현하는 프로세스
- 팀원들이 워크플로우에 익숙해질 때까지 연습

### 3단계: Hook + 서브에이전트 (2주~)

- 자동 품질 검사 Hook 활성화
- QA, 테스트 서브에이전트 활용
- 팀 맞춤 스킬/프롬프트 추가

### 4단계: 팀 전체 표준화 (4주~)

- 하네스 설정을 Git에 커밋하여 팀 전체가 동일한 환경 사용
- 프롬프트 라이브러리를 팀 공유
- 하네스 설정을 지속적으로 개선

## 자주 묻는 질문

**Q: CLAUDE.md를 Git에 커밋해야 하나요?**
> 네. 팀 전체가 동일한 AI 규칙을 사용하려면 Git에 포함해야 합니다. 단, 민감 정보(API 키, 내부 URL 등)는 포함하지 마세요.

**Q: 하네스 없이도 AI를 잘 쓰고 있는데, 꼭 필요한가요?**
> 개인 수준에서는 없이도 잘 사용할 수 있습니다. 하네스의 가치는 **팀 수준의 일관성과 안전장치**입니다. 팀원이 3명 이상이라면 도입을 권장합니다.

**Q: dev_sys_template 대신 자체 하네스를 만들어도 되나요?**
> 물론입니다. dev_sys_template은 참고용 템플릿입니다. 핵심은 CLAUDE.md + 워크플로우 규칙이며, 이 두 가지를 직접 구성해도 됩니다.

**Q: Hook이 너무 엄격하면 오히려 속도가 느려지지 않나요?**
> 그럴 수 있습니다. 처음에는 Hook을 느슨하게 설정하고, 팀이 익숙해지면 점진적으로 강화하세요. Hook은 설정으로 끄고 켤 수 있습니다.
