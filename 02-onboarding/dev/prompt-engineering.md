# 개발자용 프롬프트 엔지니어링 가이드

> 같은 AI 도구를 써도 프롬프트에 따라 결과가 크게 달라집니다. 이 가이드는 개발 업무에서 바로 쓸 수 있는 프롬프트 패턴과 템플릿을 제공합니다.

---

## 핵심 원칙

### 1. 맥락을 충분히 제공한다

AI는 당신의 프로젝트를 모릅니다 (하네스를 쓰지 않는 경우). 기술 스택, 코딩 컨벤션, 파일 구조를 명시적으로 알려주세요.

### 2. 한 번에 하나만 요청한다

"로그인 기능 만들고 테스트도 작성하고 문서화도 해줘"보다, 각각 나눠서 요청하는 것이 품질이 높습니다.

### 3. 제약 조건을 명시한다

"좋은 코드 작성해줘"는 모호합니다. "TypeScript, 함수형, 에러 핸들링 포함, 100줄 이내"처럼 구체적으로 제약하세요.

### 4. Before/After로 검증한다

AI 결과물을 기존 코드와 비교하여 의도하지 않은 변경이 없는지 확인합니다.

---

## 프롬프트 패턴

### 패턴 1: 역할 + 맥락 + 작업

가장 기본적이고 범용적인 패턴입니다.

```
[역할] 당신은 시니어 TypeScript 개발자입니다.
[맥락] 이 프로젝트는 Next.js 14 + TypeScript + Prisma ORM을 사용합니다.
[작업] 사용자 프로필 조회 API 엔드포인트를 작성해주세요.
[제약] - App Router 사용
       - 에러 핸들링 포함
       - Zod로 입력 검증
```

### 패턴 2: 예시 기반 (Few-shot)

기존 코드의 패턴을 따라야 할 때 효과적입니다.

```
아래 기존 API 엔드포인트의 패턴을 따라서 새 엔드포인트를 작성해줘.

[기존 코드]
// app/api/users/route.ts
(기존 코드 붙여넣기)

[요청]
같은 패턴으로 app/api/posts/route.ts를 작성해줘.
- GET: 전체 목록 조회 (페이지네이션)
- POST: 새 글 생성
```

### 패턴 3: 단계적 요청 (Chain of Thought)

복잡한 작업을 단계별로 나눕니다.

```
아래 순서로 진행해줘. 각 단계의 결과를 보여준 후 다음 단계로 넘어가줘.

1단계: 현재 src/auth/ 디렉토리의 구조를 분석해줘
2단계: 인증 흐름에서 개선이 필요한 부분을 식별해줘
3단계: 개선 방안을 제안해줘 (코드 변경 전)
4단계: 승인하면 코드를 수정해줘
```

## 업무별 프롬프트 템플릿

### 코드 생성

**기본 구조:**
```
[기술 스택] TypeScript + Express
[작업] 파일 업로드 미들웨어를 작성해줘.
[요구사항]
- multer 사용
- 파일 크기 제한: 10MB
- 허용 확장자: jpg, png, pdf
- 에러 시 적절한 HTTP 상태 코드 반환
[참고] 기존 미들웨어 패턴: src/middleware/auth.ts
```

**Before (모호한 프롬프트):**
```
파일 업로드 기능 만들어줘
```
→ 어떤 언어인지, 라이브러리는 뭔지, 제약은 뭔지 모름

**After (구체적 프롬프트):**
```
Express + TypeScript 프로젝트에서 multer를 사용하는 파일 업로드 미들웨어를 작성해줘.

요구사항:
- 최대 파일 크기: 10MB
- 허용 확장자: jpg, png, pdf
- 업로드 경로: /uploads/{yyyy-MM-dd}/
- 에러 처리: 파일 크기 초과 시 413, 확장자 불일치 시 415 반환
- 기존 미들웨어 패턴(src/middleware/auth.ts)과 동일한 구조로 작성
```
→ 명확한 제약과 기존 코드 참조로 일관된 결과

### 리팩토링

**기본 구조:**
```
아래 코드를 리팩토링해줘.

[현재 코드]
(코드 붙여넣기)

[목표]
- [구체적 개선 목표]
[제약]
- 외부 인터페이스(함수 시그니처, API 응답 형태)는 변경하지 않기
- 기존 테스트가 깨지지 않도록
```

**Before:**
```
이 코드 좀 깔끔하게 해줘

function processData(data) {
  // 100줄짜리 복잡한 함수
}
```

**After:**
```
아래 함수를 리팩토링해줘.

목표:
1. 하나의 큰 함수를 역할별로 분리 (단일 책임 원칙)
2. 매직 넘버를 의미 있는 상수로 추출
3. 에러 케이스를 early return으로 처리

제약:
- 함수의 입력/출력 타입은 변경하지 않기
- processData()를 호출하는 곳은 수정하지 않기
- 새로운 외부 라이브러리 추가 금지

function processData(data) {
  // 100줄짜리 복잡한 함수
}
```

### 디버깅

**기본 구조:**
```
아래 버그를 분석하고 수정해줘.

[증상] 무엇이 잘못되고 있는가
[기대 동작] 원래 어떻게 되어야 하는가
[재현 조건] 어떤 상황에서 발생하는가
[관련 코드] (코드 또는 파일 경로)
[에러 로그] (있으면 붙여넣기)
```

**Before:**
```
로그인이 안 돼요. 고쳐주세요.
```

**After:**
```
로그인 시 간헐적으로 401 에러가 발생합니다.

증상: 로그인 성공 후 /dashboard로 리다이렉트 시 401 반환
기대 동작: 로그인 성공 → 토큰 발급 → /dashboard 정상 접근
재현 조건: 5분 이상 유휴 상태 후 재로그인 시 빈번하게 발생
관련 코드: src/auth/middleware.ts, src/auth/token.ts
에러 로그:
[2024-03-15 14:23:01] ERROR: Token verification failed - TokenExpiredError: jwt expired

이 에러의 원인을 분석하고 수정 방안을 제안해줘.
코드를 수정하기 전에 원인 분석부터 보여줘.
```

### 코드 리뷰

**기본 구조:**
```
아래 코드를 리뷰해줘.

[관점] 어떤 측면을 중점적으로 봐야 하는가
[코드] (코드 또는 diff)
```

**템플릿:**
```
아래 PR의 코드를 리뷰해줘.

중점 검토 항목:
1. 보안 취약점 (SQL 인젝션, XSS, 인증 우회 등)
2. 에러 핸들링 누락
3. 성능 문제 (N+1 쿼리, 불필요한 연산 등)
4. 기존 코드와의 일관성

코드:
(diff 또는 코드 붙여넣기)

각 이슈에 대해 [심각도: 높음/중간/낮음]과 수정 제안을 포함해줘.
```

### 테스트 작성

**기본 구조:**
```
아래 함수에 대한 테스트를 작성해줘.

[테스트 프레임워크] Jest / Vitest / pytest 등
[대상 코드] (코드 또는 파일 경로)
[테스트 범위] 어떤 케이스를 커버해야 하는가
```

**템플릿:**
```
아래 함수에 대한 단위 테스트를 Vitest로 작성해줘.

대상: src/utils/validator.ts의 validateEmail() 함수

테스트 케이스:
- 정상 이메일 (기본, 서브도메인, + 태그)
- 비정상 이메일 (@없음, 도메인 없음, 공백 포함)
- 엣지 케이스 (빈 문자열, null, undefined, 매우 긴 문자열)

기존 테스트 파일 패턴: src/utils/__tests__/formatter.test.ts를 참고해서
동일한 구조(describe/it, 네이밍 패턴)로 작성해줘.
```

### 문서화

**기본 구조:**
```
아래 코드의 문서를 작성해줘.

[형식] JSDoc / README / API 문서 등
[대상] (코드 또는 파일 경로)
[독자] 이 문서를 읽을 사람
```

**템플릿:**
```
src/api/ 디렉토리의 API 엔드포인트 문서를 작성해줘.

형식: 마크다운 (README.md)
독자: 이 API를 사용할 프론트엔드 개발자
포함 항목:
- 각 엔드포인트의 URL, 메서드, 파라미터
- 요청/응답 예시 (JSON)
- 에러 코드와 의미
- 인증 방식
```

## 고급 기법

### 1. 점진적 구체화

처음에 큰 그림을 잡고, 점점 구체화합니다.

```
1회차: "사용자 인증 시스템의 아키텍처를 설계해줘"
→ 구조 확인

2회차: "이 설계에서 JWT 토큰 관리 부분을 상세하게 구현해줘"
→ 핵심 로직 구현

3회차: "토큰 갱신 로직에 대한 엣지 케이스 테스트를 작성해줘"
→ 품질 보강
```

### 2. 제약 기반 생성

원하지 않는 것을 명시하면 품질이 올라갑니다.

```
사용자 목록 API를 작성해줘.

하지 말 것:
- ORM 전체 조회 (SELECT * 금지 → 필요한 필드만)
- 동기 처리 (비동기로 처리)
- any 타입 사용 금지
- console.log 사용 금지 (logger 사용)
```

### 3. 기존 코드 참조

AI에게 "이 프로젝트의 스타일"을 가르치는 가장 효과적인 방법입니다.

```
src/services/userService.ts의 구조와 패턴을 따라서
src/services/postService.ts를 작성해줘.

특히 아래를 동일하게 유지해줘:
- 에러 핸들링 패턴
- 로깅 패턴
- 리턴 타입 구조
```

## 프롬프트 체크리스트

작성한 프롬프트를 보내기 전에 확인하세요:

- [ ] **맥락**: 기술 스택과 환경을 명시했는가?
- [ ] **구체성**: 모호한 표현("좋은", "깔끔한", "적절한") 대신 구체적 기준을 줬는가?
- [ ] **제약**: 변경하면 안 되는 것, 사용하면 안 되는 것을 명시했는가?
- [ ] **참조**: 따라야 할 기존 코드 패턴을 알려줬는가?
- [ ] **범위**: 한 번에 너무 많은 것을 요청하고 있지 않은가?
- [ ] **보안**: 민감정보(API 키, 비밀번호 등)를 프롬프트에 포함하지 않았는가?
